## 1. What this project is

A portfolio artifact for an AI strategy consultant role. The CV value is not the app, it is the demonstrated judgment: architecture decisions, cost engineering, trade-off documentation and evals. Built end to end using AI (the builder does not hand-code) with every trade-off owned and explainable. Defensibility target: L2 leaning L3.

Product vision (confirmed session 3): two-fold, not wardrobe-only. The stylist looks at the owned wardrobe first, recommends outfits from it and where a gap exists it recommends a real purchasable product with a clickable link redirecting to the retailer. A static catalogue was rejected by the builder as not solving the actual use case.

## 6. Shared attribute schema

### Tier 1: constraint-eligible
id, source (wardrobe/shopping), category, subcategory, article_type, colour_primary, formality, season, gender, image_ref, display_name. Shopping-only: price, currency, merchant, product_link. Price and merchant come structured from Serper so budget and whitelist filters stay deterministic.

### Tier 2: rich for wardrobe, LLM-inferred for shopping results
pattern, fit, material, length_attributes (sleeve/hem/rise), style_tags. Hard constraints on tier 2 fields are best-effort on shopping results. Two-tier guarantee documented in the README.

### Validation rule
Enums enforced in code. Off-spec model output coerced to null and flagged needs_review. Constraint filters never trust raw model strings. Implemented in extract_wardrobe.py. Known v1 limitation: one category per item, convertible garments get the category of how they are most often worn.

## 8. Architecture principles

1. Deterministic constraint logic in code, not prompts.
2. Stateful memory and caching in files, never in the model.
3. Extract expensive attributes once, cache, reuse.
4. Retrieval layer decoupled from data source (Serper today, Awin in v2, same interface).
5. Hard constraints only reference guaranteed-populated fields, tier 2 constraints best-effort on shopping results.
6. Every phase produces something demoable.
7. Every external call metered, capped in code and logged.
8. Model IDs are pinned from evidence (list-models against the key), never from documentation alone. Added session 4 after the deprecation saga.
